## 2. 사용자 스토리 분할 방법론 선택 (Story Breakdown Options)

사용자 스토리를 조직하고 구조화하기 위한 3가지 방법론을 제안합니다. 프로젝트 목적에 가장 적절한 방식을 선택해 주십시오.

### 옵션 A: 사용자 여정 중심 (User Journey-Based)
- **설명**: 사용자가 플랫폼에 접근하여 작업을 요청하고, SSE 스트리밍을 모니터링하며, 최종 파일을 다운로드하는 시간적 흐름(User Journey)을 따라 스토리를 분할합니다.
- **장점**: 사용자 경험(UX) 관점의 흐름이 명확하고 프론트-백엔드 간의 실시간 연동 인터랙션 누락을 방지하기 쉽습니다.
- **단점**: 기능적 독립성(예: LLM Plan 파서와 SSE 연동 등 백엔드 모듈성)을 개별 스토리로 쪼개기 모호할 수 있습니다.

### 옵션 B: 기능 중심 (Feature-Based)
- **설명**: 시스템의 주요 컴포넌트(Job 관리, LLM Plan 엔진, OpenSCAD Runner, SSE 스트리머, 스토리지)별로 스토리를 구성합니다.
- **장점**: 개발 백로그 구성 및 컴포넌트 단위의 작업 할당이 직관적입니다.
- **단점**: 사용자가 느끼는 전체적인 흐름이 단절되어 보일 수 있습니다.

### 옵션 C: 하이브리드 (Hybrid Journey + Feature) - *선택됨*
- **설명**: 사용자가 직접 경험하는 인터페이스/SSE 부분은 사용자 여정 중심으로 구성하고, 백엔드의 격리 프로세스 실행 엔진 및 LLM plan 검증기는 기능 중심으로 묶어 구성합니다.
- **장점**: 사용자 시나리오의 완결성을 유지하면서 백엔드 핵심 비기능적 구현도 독립적인 스토리로 명확히 관리할 수 있습니다.
- **단점**: 관리할 스토리 개수가 조금 늘어날 수 있습니다.

### 질문 3.2: 페르소나(Persona) 모델링
이 플랫폼을 주로 사용하거나 관리하는 역할군은 누구입니까? (다중 선택 가능)
- **A) 일반 사용자 (End-User)**: 3D 프린팅이나 CAD 설계에 전문 지식이 없지만, 자연어로 "도킹스테이션 만들어줘"처럼 쉽게 설계물을 제작하고 다운로드받기를 원하는 사용자
- **B) 전문 메이커/설계사 (Professional Maker)**: CAD 코드를 이해하며, LLM이 만든 model.scad 결과물을 리뷰하고 수정 사항을 다시 요구하는 등 정밀 제어가 필요한 사용자
- **C) 플랫폼 관리자 (Platform Admin)**: 실행 중인 Job 목록을 모니터링하고, 비정상적인 CPU/메모리/시간을 점유하는 CLI 프로세스를 제어하려는 관리자
- **X) 기타**

[Answer]: 일반 사용자 (End-User) - 3D/CAD 전문지식은 없으나 자연어로 결과물을 만들고 다운로드하고 싶은 사용자

### 질문 3.3: SSE 실시간 스트리밍 시나리오
사용자가 SSE 연결을 유지하거나 재연결할 때의 경험에서 가장 중요하게 정의하고 싶은 규칙은 무엇입니까?
- **A) 무중단 자동 재연결 및 완벽한 복구**: 클라이언트 라이브러리가 연결 유실 시 자동으로 Last-Event-ID를 실어 재연결을 시도하고, 누락 구간의 로그를 화면에 끊김 없이 이어 붙여서 보여주어야 함.
- **B) 재연결 수동 복구**: 화면 내에 "새로고침/로그 재동기화" 버튼을 두어 사용자가 필요할 때만 지난 로그를 일괄적으로 가져올 수 있게 함.
- **X) 기타**

[Answer]: 무중단 자동 재연결 및 완벽 복구 (네트워크 단절 시 클라이언트가 Last-Event-ID로 자동 복구하여 실시간 로그를 매끄럽게 표시)

---

### 질문 3.4: LLM 피드백 및 반복 설계 (Interactive Refinement)
사용자가 LLM이 생성한 초기 설계물(model.scad)을 확인한 뒤, 추가적인 자연어 입력(예: "충전 구멍을 조금만 더 키워줘")을 통해 기존 Job이나 model.scad를 수정하는 반복 시나리오를 MVP 요구사항 스토리로 고려해야 할까요?
- **A) Yes**: 단일 Job 내에서 혹은 이전 Job을 참조하여 설계를 점진적으로 개선하는 시나리오를 스토리에 포함해야 함.
- **B) No (MVP 제외)**: MVP에서는 사용자의 1회성 요청에 대해 1개의 Job이 성공/실패하는 단발성 흐름으로 한정하고, 대화형 수정은 차후 확장함.
- **X) 기타**

[Answer]: Yes - 단일 Job 내 혹은 이전 Job 참조를 통해 자연어 추가 수정(피드백)을 수용하여 설계 코드를 반복 개선하는 스토리를 포함
