# 비기능 요구사항 - Unit 4: SSE Streaming & Event Catch-up

## 범위

Unit 4의 SSE 스트리밍, 재연결 catch-up, Job별 접근 토큰에 적용할 성능, 확장성, 가용성, 보안, 신뢰성 기준을 정의합니다.

## 성능 및 확장성

### NFR-4-1: 동시 연결

- 단일 애플리케이션 인스턴스는 최대 20개의 SSE 연결을 동시에 유지해야 합니다.
- 동일 Job에 대한 다중 연결을 허용하며 각 연결은 독립적으로 이벤트를 조회합니다.
- 20개를 초과한 연결은 HTTP 503과 재시도 가능한 오류 정보로 거절해야 합니다.

### NFR-4-2: 이벤트 전달 지연

- 20개 동시 연결 부하에서 `EventLog`가 DB에 커밋된 시점부터 SSE 프레임 전송 시작까지 평균 3초 이내여야 합니다.
- 측정은 최소 100개 이벤트 표본을 사용하며, 테스트 환경과 측정 결과를 기록해야 합니다.
- polling 간격은 0.5초로 고정합니다.

### NFR-4-3: 조회 효율

- 이벤트 조회는 `job_id`, `id > last_seen_id`, `id ASC` 조건과 기존 `idx_event_logs_job_id_id` 인덱스를 사용해야 합니다.
- 한 번의 polling 조회 건수에는 상한을 두어 대량 catch-up이 다른 연결을 장시간 점유하지 않게 해야 합니다.
- DB 세션은 연결 수명 동안 점유하지 않고 각 조회와 상태 확인이 끝나면 즉시 반환해야 합니다.

## 가용성 및 신뢰성

### NFR-4-4: 연결 수명

- 한 SSE 연결의 최대 유지 시간은 10분입니다.
- 10분에 도달하면 서버는 재연결 가능한 종료 제어 이벤트를 전송하고 연결을 닫아야 합니다.
- 클라이언트는 마지막 수신 이벤트 ID를 `Last-Event-ID`로 전달해 누락 지점부터 재연결할 수 있어야 합니다.

### NFR-4-5: 전달 순서와 중복 허용

- 이벤트는 `EventLog.id` 오름차순으로 전달해야 합니다.
- 전송 보장은 at-least-once로 정의하며, 재연결 경계에서 중복 이벤트가 발생할 수 있습니다.
- 클라이언트가 SSE `id`를 기준으로 중복을 제거할 수 있도록 모든 저장 이벤트에 ID를 포함해야 합니다.

### NFR-4-6: 종료 및 오류 처리

- 종료된 Job은 저장 이벤트를 모두 전송한 뒤 `STREAM_END` 이벤트를 보내고 닫아야 합니다.
- 연결 후 DB 오류가 발생하면 가능한 경우 `STREAM_ERROR` 이벤트를 전송하고 연결을 닫아야 합니다.
- 클라이언트 연결 해제는 정상 취소로 처리하며 서버 오류 로그를 불필요하게 남기지 않아야 합니다.

## 보안

### NFR-4-7: Job별 스트림 토큰

- Job 생성 응답은 해당 Job에만 유효한 스트림 토큰을 포함해야 합니다.
- 토큰은 서버 비밀키로 서명하며 요청의 `job_id`와 서명된 Job ID가 일치해야 합니다.
- 토큰이 없거나 서명이 잘못되었거나 다른 Job의 토큰이면 HTTP 403을 반환해야 합니다.
- 토큰과 서명 비밀키는 애플리케이션 로그, 이벤트 로그, 오류 본문에 기록하면 안 됩니다.
- 서명 비교는 타이밍 공격을 줄이기 위해 상수 시간 비교 함수를 사용해야 합니다.

### NFR-4-8: 비밀키 관리

- 서명 비밀키는 환경 변수 `SSE_STREAM_TOKEN_SECRET`에서 읽어야 합니다.
- 운영 환경에서는 비밀키가 없으면 애플리케이션 시작 또는 스트림 기능 초기화에 실패해야 합니다.
- 개발 및 테스트용 기본 비밀키를 운영 설정에 포함하면 안 됩니다.

## 유지보수성 및 검증

### NFR-4-9: 설정과 관측

- 연결 상한 20개, polling 간격 0.5초, 최대 연결 시간 10분은 명명된 설정 또는 상수로 관리해야 합니다.
- 연결 시작, 정상 종료, 제한 거절, 내부 오류를 토큰 없이 구조화 로그로 남겨야 합니다.

### NFR-4-10: 필수 테스트

- 유효/무효/타 Job 토큰 검증 테스트
- `Last-Event-ID` 유효, 누락, 비숫자, 범위 밖 처리 테스트
- 이벤트 순서, catch-up, 종료 이벤트 테스트
- 10분 연결 제한은 시간을 주입하거나 가상화한 테스트
- 20개 동시 연결과 21번째 연결 거절 테스트
- 20개 연결에서 평균 전달 지연 3초 이내 성능 테스트
- 연결 해제 후 DB 세션과 연결 카운터 반환 테스트

## 범위 제외

- Redis, Kafka, RabbitMQ 등 외부 Queue/PubSub 인프라
- 다중 애플리케이션 인스턴스 간 전역 연결 수 제한
- 사용자 계정 및 Job 소유권 기반 권한 모델
- exactly-once 전송 보장
