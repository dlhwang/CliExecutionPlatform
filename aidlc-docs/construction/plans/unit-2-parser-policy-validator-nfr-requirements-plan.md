## 2. 의사결정을 위한 공개 질문지 (Open Questions)

> [!IMPORTANT]
> **Unit 2의 비기능 요구사항 및 기술 스택을 설계하기 위해 아래 질문에 답변해 주십시오.**  
> 각 질문에 대해 적절한 옵션을 선택하거나 세부 사항을 기재해 주십시오.

### 질문 1. 파싱 및 보안 검증 성능 지연 시간(Latency) 목표
LLM의 원본 마크다운 응답을 수신한 직후, 파서가 동작하여 JSON을 추출하고 보안 검증을 수행하기까지의 허용 지연 시간 목표는 무엇입니까?
* **옵션 A (지연시간 극소화)**: 100ms 이내. 로컬 인메모리 연산 위주이므로 지연을 최소화하도록 라이트웨이트 정규식과 Pydantic 매핑 가동.
* **옵션 B (정밀 검증 및 감사 로깅 우선)**: 500ms 이내. 호스트 파일시스템 I/O(존재 확인, 절대경로 검사 등) 및 로깅 적재 프로세스의 정밀성을 우선.
* **선택**: 
[Answer]: B

### 질문 2. 보안 침투(Path Traversal 등) 발생 시 보안 감사 및 로깅 수준
경로 탈출 공격이나 비인가 CLI 도구 실행 시도가 감지되어 예외가 유발되었을 때, 시스템 내부의 감사 및 보안 로깅 수준을 어떻게 설계할까요?
* **옵션 A (상세 감사 로깅 및 DB 적재)**: 시스템 내부 경고(`warning` 레벨) 로깅을 수행할 뿐만 아니라, `event_logs` 테이블에 위반 일시 및 해당 Job ID를 `SYSTEM` 혹은 `SECURITY_ALERT` 타입의 이벤트 로그로 영구 적재하여 관리자가 모니터링할 수 있도록 설계.
* **옵션 B (API 에러 반환 및 파일 로깅)**: 단순 백엔드 파일 로그에만 `warning`으로 기록하고 DB에는 기록하지 않음 (오인 유발을 방지하기 위해 노이즈 최소화).
* **선택**: 
[Answer]: A

### 질문 3. LLM JSON 구문 오류 시 복구(재시도)를 위한 예외 아키텍처
LLM이 JSON 형식을 맞추지 못해 파싱에 실패한 경우, 즉시 실패처리하지 않고 LLM에게 재수정을 요청(재시도 피드백 루프)하기 위해 파서 레벨에서 전용의 "재시도 가능 예외(Retryable Exception)" 타입을 정의하여 상위 오케스트레이터로 전달할까요?
* **옵션 A (재시도 예외 구조 지원)**: `LLMPlanRetryableException` 과 같은 전용 예외 클래스를 만들어, JSON 오류 시 이를 던져 오케스트레이터가 재시도 루프를 탈 수 있는 논리적 인터페이스를 설계함.
* **옵션 B (단순 실패 처리)**: 재시도 가능성을 열어두지 않고 단순 유효성 에러(`ParserError`)로 퉁쳐서 즉시 실패(`FAILED`) 상태로 귀결시킴.
* **선택**: 
[Answer]: A
